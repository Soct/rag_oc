Projet Python • FastAPI • RAG local

RAG OpenAgenda pour recommander des evenements

Ce projet construit un moteur de recherche semantique sur des evenements OpenAgenda en Ile-de-France. Il collecte les donnees, fabrique un index FAISS, interroge Mistral pour les embeddings et la generation, puis expose le tout via une API FastAPI testable en local ou dans Docker.

3 Endpoints principaux exposes par l'API : /health, /ask, /rebuild.
4 Briques majeures : collecte, indexation, service RAG, exposition HTTP.
100% Flux local et reproductible avec uv, Docker et une documentation Swagger prete pour la demo.

Stack technique

  • OpenAgenda pour la source evenementielle.
  • FAISS pour l'index vectoriel et le retrieval.
  • Mistral pour les embeddings et la reponse finale.
  • FastAPI pour l'API REST et la documentation interactive.
  • uv pour l'environnement Python et l'image Docker.

Objectif et promesse

Transformer une base d'evenements volumineuse et heterogene en experience de question-reponse exploitable pour une demo technique claire.

Probleme

Une recherche classique par mots-cles ne suffit pas quand les descriptions sont longues, variees et parfois mal structurees.

Solution

Le projet convertit les textes en embeddings, retrouve les passages pertinents, reconstruit un contexte, puis genere une reponse naturelle.

Resultat

Une API capable de repondre a des questions comme atelier a Paris ce week-end ou sortie famille prochainement.

Pipeline du projet

Le repo est organise comme un enchainement simple et lisible : ingestion, preparation, indexation, interrogation.

  1. 01
    Collecte OpenAgenda

    scripts/fetch_openagenda_events.py appelle l'API OpenAgenda, filtre la zone cible, normalise les champs utiles et ecrit un fichier JSONL ou JSONL.gz.

  2. 02
    Construction du corpus

    Le champ document devient la base du chunking. Le manifeste de collecte garde aussi la fenetre temporelle et les informations de run.

  3. 03
    Embeddings et FAISS

    scripts/build_faiss_index.py decoupe le texte, appelle Mistral pour les embeddings et produit data/faiss/openagenda.index avec sa metadata associee.

  4. 04
    Question-reponse RAG

    src/rag_oc/rag_service.py cherche les chunks proches, filtre les evenements passes, reconstruit le contexte et demande au modele de chat la reponse finale.

  5. 05
    Exposition API

    src/rag_oc/api.py sert le moteur via FastAPI avec Swagger et des scenarios de test preconfigures dans l'interface.

Architecture et execution

Le projet reste volontairement compact. La logique est separee entre scripts, service RAG et couche HTTP.

Fichiers clefs

  • scripts/fetch_openagenda_events.py : ingestion OpenAgenda.
  • scripts/build_faiss_index.py : chunking, embeddings, index FAISS.
  • src/rag_oc/rag_service.py : coeur metier du retrieval et de la generation.
  • src/rag_oc/api.py : endpoints FastAPI et scenarios OpenAPI.
  • scripts/evaluate_rag.py : base d'evaluation automatique avec Ragas.

API et scenarios de demo

L'API est pensee pour une demonstration rapide : des endpoints simples, Swagger sur /docs et des exemples directement rejouables.

Endpoints

GET/health
POST/ask
POST/rebuild

L'endpoint /ask renvoie une reponse, les sources retenues et le contexte assemble. L'endpoint /rebuild relance la construction de l'index puis recharge le service si necessaire.

Scenarios de test

  • Atelier a Paris ce week-end Valide la recherche semantique sur une activite creative a horizon court.
  • Evenements famille prochainement Verifie la restitution d'options plus larges avec un contexte plus riche.
  • Exposition gratuite en Ile-de-France Teste la capacite a combiner sujet, cout et localisation.

Evaluation, limites, suites

Le projet dispose deja d'une base de tests et d'un squelette pour l'evaluation automatique, mais il reste de la marge pour l'industrialisation.

Ce qui est deja couvert

  • Tests unitaires sur la collecte, l'indexation, le service RAG et le contrat API.
  • Script scripts/evaluate_rag.py prevu pour brancher Ragas et comparer les reponses.
  • Verification locale via API et Swagger.

Limites actuelles

  • Dependance a Mistral pour les embeddings et la generation.
  • Temps de rebuild d'index potentiellement significatif.
  • Qualite sensible a la qualite des donnees source et au retrieval.

Pistes d'evolution

  • Reranking des resultats.
  • Filtres metier explicites par date, lieu et categorie.
  • Durcissement de /rebuild et deploiement plus industrialise.
Resume

Un POC RAG local, proprement structure, demonstrable vite, et deja suffisamment outille pour servir de base a une evolution produit.

OpenAgenda → FAISS → Mistral → FastAPI
